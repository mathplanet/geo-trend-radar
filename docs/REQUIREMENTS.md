# GEO Trend Radar — 요구사항 정의서

> 작성일: 2026-07-16 · 상태: 빌드 착수 준비 완료 (1단계 프로토타입 검증됨)

## 1. 개요

GEO/SEO 트렌드를 자동으로 센싱해 **팀이 함께 보는 웹 뉴스레터/대시보드**를 만드는 프로젝트.
여러 SEO/AI검색 매체의 RSS를 주기적으로 수집 → 키워드로 유의미한 글만 필터 →
Claude로 요약·인사이트 생성 → 웹 대시보드에 주간 다이제스트로 발행한다.

## 2. 배경 & 문제

- GEO(Generative Engine Optimization)는 변화가 빠르고 출처가 파편화되어 있어,
  개인이 매일 여러 매체를 수동으로 따라가기 어렵다.
- 팀 내부에 트렌드를 **한 곳에서, 요약된 형태로, 정기적으로** 공유하는 채널이 없다.
- 목표는 "수동 큐레이션"을 "자동 파이프라인 + 사람의 최종 감수"로 대체하는 것.

## 3. 목표 & 성공 기준

| 목표 | 성공 기준 (측정 가능) |
|------|----------------------|
| 유의미한 글만 자동 수집 | 주간 다이제스트에서 팀이 "관련 없음"으로 판단하는 글 비율 < 20% |
| 정기 발행 자동화 | 사람 개입 없이 주 1회 대시보드가 자동 갱신됨 |
| 요약 품질 | 각 글의 3줄 요약 + "클라이언트 시사점"이 원문 안 읽어도 이해되는 수준 |
| 팀 접근성 | 팀원이 URL 하나로 접근, 아카이브 열람 가능 |

## 4. 범위

### In scope (v1)
- RSS 기반 소스 수집 (12개 소스, 확장 가능)
- 키워드 가중치 + 출처 등급별 threshold 필터링
- URL 해시 기반 중복 제거
- Claude 기반 요약·주제 클러스터링·인사이트 생성
- 웹 대시보드 (이번 주 + 주차별 아카이브)
- 격일 수집 / 주간 발행 자동 스케줄링

### Out of scope (v1 — 이후 고려)
- RSS 없는 소스의 본문 크롤링 (`type: crawl`) — v2
- 시맨틱 중복 제거(임베딩/pgvector) — 지금은 URL 해시로 충분
- 팀 로그인/인증 — v1은 비공개 URL, 필요 시 나중에 추가
- 이메일 발송, 코멘트/북마크 등 협업 기능
- 다국어 요약 (v1은 영문 소스 → 한글 요약 여부는 5번 결정사항 참고)

## 5. 사용자 & 시나리오

- **주 사용자**: GEO/SEO 컨설팅 팀 (컨설턴트 다수)
- **시나리오**: 매주 월요일 아침, 팀원이 대시보드 URL을 열어 지난 주 주요 트렌드 3~5건과
  주제별 정리, 각 글의 시사점을 5분 안에 훑는다. 필요 시 원문 링크로 이동.

## 6. 기능 요구사항 (FR)

| ID | 요구사항 |
|----|----------|
| FR-1 | `sources.yaml`의 RSS 피드를 수집한다. 소스는 파일 편집만으로 추가/삭제 가능해야 한다. |
| FR-2 | `keywords.yaml`의 키워드+가중치로 각 글의 relevance_score(가중치 합)를 계산한다. |
| FR-3 | 출처 등급(tier)별 threshold를 적용해 score ≥ threshold인 글만 채택한다. |
| FR-4 | URL 해시로 중복 글을 제거한다 (동일 글 재적재 방지). |
| FR-5 | 채택된 글을 DB에 적재한다 (원문 메타 + 요약 필드). |
| FR-6 | 주간 배치에서 Claude가 관련도 재평가·3줄 요약·주제 클러스터·클라이언트 시사점을 생성한다. |
| FR-7 | 웹 대시보드가 이번 주 다이제스트와 주차별 아카이브를 렌더한다. |
| FR-8 | 격일 수집·주간 발행이 cron으로 자동 실행된다. |

## 7. 비기능 요구사항 (NFR)

- **비용**: LLM 호출은 주 1회 배치로 제한 (건별 상시 호출 금지). 전 서비스 무료 티어 우선.
- **신뢰성**: 개별 피드 실패가 전체 파이프라인을 중단시키지 않아야 한다 (소스별 격리).
- **유지보수성**: 소스·키워드·threshold는 코드 수정 없이 설정 파일(yaml)로 관리.
- **관측성**: 수집 배치는 소스별 (총엔트리 → 통과 건수)를 로그로 남긴다.
- **멱등성**: 배치를 여러 번 돌려도 중복 적재가 없어야 한다.

## 8. 시스템 아키텍처

```
[격일 cron]  GitHub Actions · collect (Python)
   RSS 수집 → 키워드 필터 → 등급별 threshold → URL해시 중복제거 → Supabase.items upsert

[주간 cron]  GitHub Actions · publish (Python)
   Supabase 최근 7일 조회 → Claude(관련도+요약+클러스터+시사점)
   → items 업데이트 + digests 저장

[상시]  Vercel · Next.js 대시보드
   Supabase 조회 → 이번 주 다이제스트 / 아카이브 렌더
```

**서비스 역할 분담**

| 서비스 | 역할 | 선정 이유 |
|--------|------|-----------|
| GitHub Actions (Python) | 수집·요약 엔진 | 실행시간 제한 없음, Python 스크래핑 생태계. 무거운 작업에 적합 |
| Supabase (Postgres) | 데이터 + (선택)인증 | URL해시 unique로 중복 차단, SQL 검색/필터, 무료 티어 |
| Vercel (Next.js) | 웹 대시보드 | Supabase SSR 렌더. **주의: Hobby는 비상업 라이선스 → 회사용은 Pro 또는 Cloudflare Pages 검토** |
| Anthropic Claude API | 요약·인사이트 | 주간 배치에서만 호출 |

> 핵심 설계 결정: **무거운 작업(스크래핑+LLM)은 Vercel 서버리스가 아니라 GitHub Actions에서** 돌린다.
> Vercel 서버리스 타임아웃(Hobby 10초/Pro 60초)으로 다건 요약이 실패하는 것을 피하기 위함.

## 9. 데이터 모델

상세 스키마(SQL 포함)는 [`REFERENCE.md`](REFERENCE.md#db-스키마) 참조. 요약:

- `sources` — 소스 목록 (name, feed_url, tier)
- `items` — 수집 글 (url_hash **UNIQUE**, 메타 + relevance_score + summary + cluster + insight)
- `digests` — 주차별 종합본 (week, headline_items, created_at)

## 10. 스케줄링

| 배치 | 주기 | 실행 | 내용 |
|------|------|------|------|
| collect | 격일 (예: 홀수일 09:00 KST) | GitHub Actions cron | 수집·필터·적재 (LLM 미사용, 무료) |
| publish | 주 1회 (예: 월 08:00 KST) | GitHub Actions cron | 요약·인사이트·다이제스트 발행 |

> cron은 UTC 기준으로 작성해야 함에 주의 (KST = UTC+9).

## 11. 산출물 (Deliverables)

1. 동작하는 GitHub 레포 (collect / summarize / build 스크립트 + 워크플로우 2개)
2. Supabase 프로젝트 (스키마 적용)
3. Vercel 배포된 대시보드 URL
4. 소스/키워드 설정 파일 (팀이 유지보수)
5. 운영 README (실행·확장 방법)

## 12. 빌드 마일스톤

품질을 단계별로 검증하며 진행 (상세 절차는 [`BUILD-GUIDE.md`](BUILD-GUIDE.md)):

1. **수집·필터 (완료된 프로토타입 제공)** — 로컬에서 필터 품질 확인
2. **요약** — `summarize.py`로 요약·인사이트 품질 확인 (로컬 JSON 기준)
3. **저장소 전환** — 로컬 JSON → Supabase
4. **대시보드** — Next.js → Vercel 배포
5. **자동화** — GitHub Actions cron 2개 연결

## 13. 미결 결정사항 (빌더가 착수 전 확정)

| # | 결정 필요 | 기본 제안 |
|---|-----------|-----------|

| D-4 | 발행 요일/시간 | 예: 월요일 오전 (팀 합의) |
